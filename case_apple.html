[iPadOS] Captive Portal Session Bleed & Cross-Service Hijacking

Credited As: Erik Sabina (Develop Aguascalientes)

Vulnerability Type: Sandbox Escape / Session Hijacking

Program: Apple Security Bounty

Executive Summary

A critical vulnerability was discovered in the iPadOS captive portal (Wi-Fi network login screen) that allows a sandboxed authentication session to "bleed" out and grant unrestricted access to the underlying browser. This flaw allows an attacker with physical access to a device at the "Activation Lock" screen to bypass all restrictions and gain a persistent, authenticated session. This session can then be used to hijack the owner's identity across a multitude of high-value, unrelated third-party services, effectively taking over their digital life.

Technical Details: The "Session Bleed" Anomaly

The captive portal is intended to be an isolated, temporary environment for network authentication only. However, after authenticating to the network, it was found that the session's context was not properly contained. By manually navigating to a website like Google.com, the portal window transforms into a fully functional, albeit chromeless, browser. The authenticated Google session from this window is then shared with other services that use Google for OAuth, leading to a full cross-service hijack.

Confirmed Exploitation Vectors

Once the initial bypass was achieved, the persistent session was successfully used to authenticate and take control of the following services:

  • Apple Ecosystem: Full access to Apple ID management and iCloud services.
  • Google Ecosystem: Complete access to Google Account, Gmail, and Google Drive.
  • Development Platforms: Authenticated into GitHub and Replit, gaining access to source code and remote shell environments.
  • AI & Data Services: Authorized ChatGPT to access and exfiltrate files from the user's Google Drive.

The "Coup de Grâce": Remote Desktop & Network Bridge

The most critical escalation of this vulnerability was achieved by using the hijacked session to install and operate **Chrome Remote Desktop**. This allowed the locked iPad to take full control of an external machine, rendering all on-device policies irrelevant and effectively turning the iPad into a secure bridge to launch advanced tools against the local network it was connected to.
