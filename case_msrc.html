[Cross-Platform] AI-Assisted gcloud CLI Session Hijacking

Credited As: Develop Aguascalientes

Vulnerability Type: Security UI Spoofing / Session Hijacking

Programs: Google Cloud VRP, Microsoft MSRC

High-Confidence Hypothesis

This report details a novel, sophisticated, cross-platform attack vector that allows for the hijacking of Google Cloud (gcloud) CLI sessions. The method leverages third-party AI chatbot interfaces (e.g., Microsoft Copilot) to act as a **"blind proxy"** for malicious authentication, bypassing the AI's security filters and the user's consent.

Hypothesized Attack Chain

  1. Initial Compromise: The user's environment is compromised, likely via a trojanized application authorized through a deceptive OAuth screen.
  2. Credential Exfiltration: The malware exfiltrates a time-sensitive, single-use authentication secret (e.g., a DCB_ASSOCIATION string) from the victim's account.
  3. The "Blind Proxy" Exploit: The attacker pastes this secret string into a third-party AI chatbot interface. The frontend of the AI platform renders a functional gcloud login QR code on the attacker's screen.
  4. The "AI Blind Spot": The AI's security filters are bypassed, as the AI backend is unaware of the visual QR code being rendered; it only processes the text string, which does not appear malicious on its own.
  5. Session Hijacking: The attacker scans the QR code, successfully authenticating a new, malicious gcloud CLI session on their own machine with the victim's credentials, gaining full control of their GCP resources.

The "AI Blind Spot" Vulnerability

The core of this vulnerability is that the gcloud authentication flow is susceptible to having its QR codes generated on a trusted, third-party platform where the underlying security AI is blind to the final visual output. When asked what it is showing, the AI confirms it is only aware of the text string, not the QR code its own interface has displayed. This allows an attacker to bypass user consent and security checks that would occur during a normal, direct login flow.

Evidence & Proof-of-Concept Limitations

The primary evidence for this hypothesis comes from a saved chat log where the malicious DCB_ASSOCIATION string, exfiltrated from a compromised Google Account, was rendered as a functional login QR code.

  • PoC Non-Reproducibility: It is critical to note that this specific PoC is no longer reproducible. The token string was time-sensitive and has expired, which is the expected behavior for a single-use authentication credential.
  • Expired Token as Evidence: This "before and after" state serves as strong evidence that the string was a live, functional token used in the attack. The fact that it expired confirms its nature as a legitimate, temporary credential.
